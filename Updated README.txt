WHAT THIS PROJECT DOES
------------------------
This pipeline takes any highway traffic video as input and automatically:
  1. Detects and tracks all vehicles (cars, trucks, buses) using YOLOv8
  2. Auto-detects lane count from white lane markings in the video
  3. Auto-computes carriageway width from road_type + calibration points
  4. Places lane boundaries using camera calibration (homography)
  5. Extracts traffic parameters per lane (flow, speed, headway, composition)
  6. Generates a VISSIM 2026-compatible .inpx simulation file
  7. Creates both carriageways (right from video, left mirrored)

The output .inpx file opens directly in PTV VISSIM 2026 with 0 errors.

================================================================================
  HOW PHASE 1 WORKS (READ then GENERATE)
================================================================================

  Phase 1 does TWO things in sequence:

  STEP 1 — READS highway.yaml:
    → reads road_type  (you set this before running)
    → reads video.path (you set this before running)

  STEP 2 — GENERATES highway.yaml (overwrites):
    → auto-detects video FPS, resolution, frame count
    → opens frame picker window for 4 reference points
    → auto-assigns real-world coordinates from road_type:
        P1.real = [0.0,         0.0 ]  ← always origin
        P2.real = [road_width,  0.0 ]  ← auto from road_type
        P3.real = [road_width, 20.0 ]  ← auto from road_type
        P4.real = [0.0,        20.0 ]  ← always same
    → writes complete config to highway.yaml

  This is why you set road_type BEFORE running Phase 1.

================================================================================
  HOW TO CLICK REFERENCE POINTS (Phase 1)
================================================================================

  When the frame picker opens, click in this exact order:

    P1 → Bottom-LEFT  solid white line  (near camera, left edge)
    P2 → Bottom-RIGHT solid white line  (near camera, right edge)
    P3 → Top-RIGHT    above P2 position (far from camera)
    P4 → Top-LEFT     above P1 position (far from camera)

  Rules:
    → P1 and P2 should be near the BOTTOM of the frame
    → P3 and P4 should be further UP the frame
    → P3 x-value MUST be greater than P4 x-value
    → Focus on ONE carriageway only (the one with traffic toward camera)

  About P3 and P4 (perspective note):
    Due to perspective narrowing, the road appears narrower at the top.
    P3 and P4 may fall slightly OUTSIDE the visible white lines at the
    top of the frame. This is completely correct and expected.
    What matters is that P3 is directly above P2's real-world position
    and P4 is directly above P1's real-world position.

  Press Q when all 4 points are clicked.

================================================================================
  ROBUSTNESS — CLICKING ACCURACY
================================================================================

  The system is robust to minor clicking errors. Verified test results:

    Original P2 pixel x = 1179  →  Width = 11.0m, Lane = 3.67m  ✓
    Shifted  P2 pixel x = 1208  →  Width = 11.0m, Lane = 3.67m  ✓
    (+29 pixel shift = ~1.5% of 1920px frame width)

  Real-world width is determined by road_type, NOT by exact pixel
  positions. Minor clicking inaccuracy (~20-30px) has ZERO effect on
  computed lane widths and all traffic parameters.

  Automatic validation: Phase 3 verifies lane width is 3.0-4.5m
  and warns if not — catching any major clicking errors early.

================================================================================
  HOW AUTO LANE DETECTION WORKS
================================================================================

  Phase 3 automatically counts lanes:

  Step 1: Scan horizontal strip at 75% frame height
  Step 2: Find brightness peaks = white dashed lane markings
          (only within calibrated carriageway pixel range)
          (threshold = mean + 1.0 x std, min 80px peak spacing)
  Step 3: Lane count = peaks + 1
          Example: 2 peaks = 3 lanes
  Step 4: Cross-validate: lane_width = total_width / count
          Must be 3.0-4.5m (standard highway range)
  Step 5: Clamp to 2-5 lanes

  If auto-detection fails (worn or unclear markings):
    Override in highway.yaml:
      lanes:
        count: 3    ← replace "auto" with actual number

================================================================================
  HOW AUTO WIDTH WORKS
================================================================================

  Phase 3 reads width from calibration points:

    total_width = abs(P2.real_x - P1.real_x)

  P2.real_x is set automatically by Phase 1 from road_type:
    road_type: motorway_3lane → P2.real_x = 11.0
    road_type: motorway_2lane → P2.real_x = 7.0

  So the user never needs to manually calculate or type a width value.
  Just set road_type correctly and the rest is automatic.

================================================================================
  ANSWERING COMMON EVALUATION QUESTIONS
================================================================================

  Q: Will it work on a new video automatically?
  A: Yes. Change 2 lines in config + click 4 points (~7 min total).
     Lane count, width, parameters and VISSIM file are all automatic.

  Q: How is lane count detected?
  A: By scanning for white dashed lane markings in a horizontal strip
     of the frame and counting brightness peaks. 2 peaks = 3 lanes.
     No manual input required.

  Q: How is carriageway width determined?
  A: Set road_type in config (e.g. motorway_3lane = 11.0m).
     Phase 1 automatically assigns this width to P2's real coordinate.
     Phase 3 reads it back. No separate width calculation needed.

  Q: What if someone clicks reference points slightly wrong?
  A: Tested: 29px shift (~1.5% of frame) = zero impact on parameters.
     Width is set by road_type, not pixel accuracy.
     Phase 3 validates lane width is 3.0-4.5m and warns if wrong.

  Q: Why only right carriageway from video?
  A: Right carriageway has vehicles moving toward camera, giving longer
     tracking trajectories and more accurate parameters. Left carriageway
     uses mirrored parameters — standard practice in traffic simulation
     studies when only one direction is clearly observable.

  Q: How does the VISSIM file get generated without a VISSIM license?
  A: The .inpx file is XML. We inject our data into a valid VISSIM 2026
     template. No license needed to generate the file — only to run it.
